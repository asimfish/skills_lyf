# Scholar v2.5.0 — Weekly Lecture Production

> Technical documentation for the features implemented on `feature/lecture-output`.
> Generated from branch analysis on 2026-01-29.

---

## Overview

v2.5.0 makes `/teaching:lecture` production-ready for weekly semester use by adding six features across output management, iterative refinement, week-over-week continuity, quality validation, and auto-preview.

**Primary user story:** An instructor preparing weekly lectures for STAT 545 can generate, refine, and validate lecture notes with minimal friction, maintaining consistent quality across a full semester.

---

## Command Modes

The `/teaching:lecture` command operates in three distinct modes based on the flags provided:

```
/teaching:lecture <topic> [options]         → Generation mode (default)
/teaching:lecture --refine=<path> ...       → Refinement mode
/teaching:lecture --check=<path> ...        → Validation mode
```

### Mode Selection Priority

```
--check present?  → Validation mode (exits with 0 or 1)
--refine present? → Refinement mode
else              → Generation mode
```

## Feature Reference

### F1: Output Directory (`-output-dir`)

Controls where generated files are written and how filenames are constructed.

**Flags:**

| Flag                | Type    | Default   |
| ------------------- | ------- | --------- |
| `--output-dir PATH` | string  | `.` (cwd) |
| `--force`           | boolean | `false`   |

**Filename conventions:**

| Context                    | Pattern                | Example             |
| -------------------------- | ---------------------- | ------------------- |
| With `--from-plan=week08`  | `week{NN}-{slug}.qmd`  | `week08-rcbd.qmd`   |
| Without `--from-plan`      | `lecture-{slug}.qmd`   | `lecture-anova.qmd` |

**Behavior:**

- Directory is auto-created if it doesn't exist (when `--output-dir` specified)
- Existing files prompt for confirmation; `--force` skips the prompt
- Week numbers are zero-padded to 2 digits (week 8 → `08`)

**Module:** `src/teaching/utils/slugify.js`

```
slugify(str)
  → lowercase → strip non-alphanumeric to hyphens → trim leading/trailing hyphens
  → returns 'untitled' for empty/falsy input or all-special-character input

generateLectureFilename(options)
  → uses slugify() on topic, applies week-number padding from fromPlan
```

### F2: Generation Metadata (Provenance)

Every generated `.qmd` file includes invisible provenance metadata as YAML comments inside the frontmatter. These comments are ignored by Quarto but readable by Scholar for tracking and refinement.

**Schema:**

```yaml
---
title: "Randomized Complete Block Design"
# --- Scholar Generation Metadata --
# generated: 2026-02-03T14:30:00Z
# scholar_version: 2.5.0
# prompt_template: lecture-notes
# config_source: .flow/teach-config.yml
# lesson_plan: week08                    # optional
# teaching_style: formal / active-learning  # optional
# generation_time: 45.2s
# sections: 8
# --
format:
  html: { toc: true }
---
```

**Design decision:** YAML comments (not frontmatter keys) to avoid interfering with Quarto rendering. The `# --` delimiters allow programmatic extraction.

**Module:** `src/teaching/formatters/quarto-notes.js`

```
generateProvenanceComments(provenance)
  → returns multi-line string of # comments to embed in frontmatter
```

**Module:** `src/teaching/generators/lecture-notes.js`

```
getScholarVersion()   → reads version from package.json
getPromptTemplateName() → returns the template identifier
```

### F3: Section-Level Refinement (`-refine`)

Regenerates specific sections of an existing lecture without redoing the full document.

**Flags:**

| Flag                   | Type   | Required                              |
| ---------------------- | ------ | ------------------------------------- |
| `--refine PATH`        | string | Yes (for refinement mode)             |
| `--section TITLE`      | string | No (omit for full-lecture refinement) |
| `--instruction TEXT`   | string | Yes (required with `--refine`)        |

**Two refinement modes:**

| Mode    | Trigger                  | Behavior                                       |
| ------- | ------------------------ | ---------------------------------------------- |
| Section | `--refine` + `--section` | Replace one section, preserve everything else  |
| Full    | `--refine` only          | Apply instruction globally, preserve structure |

**Section matching algorithm:**

```
1. slugify(query) and slugify(sectionTitle) for both
2. Try exact slug match first
3. Fallback: either slug contains the other (fuzzy)
4. If no match: throw error listing available sections
```

**Refinement pipeline:**

```
Parse .qmd → Match section → Build prompt with context → AI generate → Splice → Update provenance → Write
```

**Context in section prompts:** The refiner includes the previous and next section titles/summaries so the AI can maintain continuity.

**Provenance update:** After each refinement, a record is appended:

```yaml
# refined: "Worked Examples" on 2026-02-05
```

For full-lecture refinement:

```yaml
# refined: full lecture on 2026-02-05
```

**Fallback (no AI):** When no API key is available, the refiner appends an HTML comment with the instruction instead of generating replacement content.

**Module:** `src/teaching/generators/lecture-refiner.js`

```
refineLecture(options)            → orchestrates full refinement flow
buildSectionRefinePrompt(...)     → prompt with section content + context + instruction
buildFullRefinePrompt(...)        → prompt with full lecture body + instruction
```

**Module:** `src/teaching/utils/qmd-parser.js`

```
parseQmdContent(content)   → { frontmatter, sections[], lines[], raw, preamble }
parseQmdFile(filePath)     → same, reading from disk
matchSection(sections, query) → matched section or null
flattenSections(sections)  → flat array including all subsections
replaceSection(lines, section, newContent) → updated document string
listSections(sections)     → formatted string for error messages
extractProvenance(frontmatter) → provenance object or null
appendRefinementRecord(frontmatter, sectionTitle, date) → updated frontmatter
```

#### QMD Parser Details

The parser builds a hierarchical section tree from markdown headings:

- **Frontmatter extraction:** Finds content between first pair of `---` delimiters
- **Heading detection:** Regex `^(#{1,6})\s+(.+?)(?:\s+\{#([^}]+)\})?$`
- **Code block awareness:** Tracks fenced code blocks (`` ``` ``) and skips headings inside them
- **Section tree:** Headings at deeper levels become subsections of the preceding shallower heading
- **Cross-reference IDs:** Captures Quarto `{#sec-name}` attributes from headings

**ParsedSection shape:**

```javascript
{
  title: string,         // "Core Concepts"
  slug: string,          // "core-concepts"
  level: number,         // 2 for ##, 3 for ###
  startLine: number,     // 0-based line index
  endLine: number,       // exclusive
  content: string,       // full section text including heading
  body: string,          // section text excluding heading line
  headingId: string,     // Quarto cross-ref ID or ""
  subsections: ParsedSection[]
}
```

### F4: Week-over-Week Continuity (`-context`)

Injects previous week content summaries into the generation prompt so lectures naturally reference earlier material.

**Flag:**

| Flag             | Type          | Default |
| ---------------- | ------------- | ------- |
| `--context SPEC` | string/number | off     |

**Values:**

- `--context=previous` — load last 3 weeks (default window)
- `--context=1` — load only last week
- `--context=5` — load last 5 weeks

**Context loading:**

1. Parse current week number from `--from-plan=weekNN`
2. Load lesson plans for weeks N-1, N-2, ..., N-count
3. Extract per week: topic, learning objectives, key concepts
4. Cap at ~500 tokens per week (~1500 total for 3-week default)
5. Inject as prompt context (not full lecture text)

**Graceful degradation:**

- Week 1: no context loaded
- Week 2: only week 1
- Week 3: weeks 1-2
- No synthetic fallback (no course overview injection)

**Module:** `src/teaching/utils/lesson-plan-loader.js`

```
loadPreviousWeekContext(options)      → array of { week, topic, objectives, keyConcepts }
formatPreviousWeekContext(contexts)   → formatted string for prompt injection
```

### F5: Coverage Validation (`-check`)

Validates that a generated lecture covers all lesson plan objectives and topics.

**Flags:**

| Flag               | Type   | Required                       |
| ------------------ | ------ | ------------------------------ |
| `--check PATH`     | string | Yes                            |
| `--from-plan WEEK` | string | Yes (required with `--check`)  |

**Algorithm:**

```
1. Parse the .qmd lecture into sections
2. Load the lesson plan objectives and topics
3. For each objective/topic:
   a. Extract keywords (strip stop words + Bloom taxonomy verbs)
   b. Search all section content for keyword matches
   c. Threshold: section matches if it contains >= ceil(keywords/2) of the keywords
4. Report coverage percentage and per-item results
```

**Output format:**

```
Coverage Report: week08-rcbd.qmd
vs content/lesson-plans/week08.yml
───────────────────────────────────────────────────

Learning Objectives:
  [x] LO-8.1: Understand RCBD design     (Sections 2, 3)
  [x] LO-8.2: Implement in R             (Section 5)
  [ ] LO-8.3: Interpret ANOVA table       (NOT FOUND)

Coverage: 2/3 (67%) - 1 gap(s) found
```

**Exit code:** 0 if 100% coverage, 1 otherwise (useful for CI).

**Stop words:** Includes standard English stop words plus Bloom's taxonomy verbs (`understand`, `apply`, `analyze`, `evaluate`, `create`, `identify`, `describe`, `explain`, `demonstrate`, `compare`, `interpret`).

**Module:** `src/teaching/validators/lecture-coverage.js`

```
validateCoverage(options)      → CoverageReport
formatCoverageReport(report)   → formatted terminal string
extractKeywords(text)          → string[] (lowercase, no stop words)
findMatchingSections(keywords, sectionContents) → string[] of titles
```

---

### F6: Auto-Preview (`-open`)

Launches Quarto preview after generation or refinement.

**Flag:**

| Flag     | Type    | Default |
| -------- | ------- | ------- |
| `--open` | boolean | `false` |

**Behavior:**

1. Check if `quarto` CLI is available
2. On macOS: use `execFileSync('osascript', ...)` for reliable app launching
3. On other platforms: use `spawn('quarto', ['preview', file], { detached: true })`
4. Process is detached so the parent can exit

**Module:** `src/teaching/utils/preview-launcher.js`

```
launchPreview(filePath, options) → { success: boolean, message: string }
isQuartoAvailable()             → boolean
```

## Architecture

### Module Map

```
src/teaching/
  utils/
    slugify.js              ← F1: Filename generation
    qmd-parser.js           ← F3: Section parsing for refinement
    preview-launcher.js     ← F6: Quarto preview launch
    lesson-plan-loader.js   ← F4: Previous week context (new functions)
    dry-run.js              ← Updated to use shared slugify
  generators/
    lecture-notes.js        ← F2/F4: Provenance tracking + context injection
    lecture-refiner.js      ← F3: Section/full refinement
  formatters/
    quarto-notes.js         ← F2: Provenance comments in frontmatter
  validators/
    lecture-coverage.js     ← F5: Coverage validation
```

### Data Flow — Generation Mode

```
User Input
  ↓
Parse args → options { topic, fromPlan, outputDir, context, ... }
  ↓
Load config (.flow/teach-config.yml)
  ↓
Load lesson plan (if --from-plan)
  ↓
Load previous week context (if --context)  ← F4
  ↓
Generate outline → Generate sections (AI)
  ↓
Validate schema
  ↓
Format as .qmd + inject provenance comments  ← F2
  ↓
Generate filename (slugify)  ← F1
  ↓
Write to --output-dir  ← F1
  ↓
Launch quarto preview (if --open)  ← F6
```

### Data Flow — Refinement Mode

```
--refine=<path> --section="Title" --instruction="..."
  ↓
parseQmdFile(path)
  ↓
matchSection(sections, "Title")
  ↓
buildSectionRefinePrompt(section, instruction, prevSection, nextSection)
  ↓
AI generate replacement
  ↓
replaceSection(lines, section, newContent)
  ↓
appendRefinementRecord(frontmatter, "Title", date)
  ↓
writeFileSync(path, updatedContent)
  ↓
Launch quarto preview (if --open)
```

### Data Flow — Validation Mode

```
--check=<path> --from-plan=week08
  ↓
parseQmdFile(path)     → sections[]
loadLessonPlan(week08) → objectives[], topics[]
  ↓
For each objective/topic:
  extractKeywords(description)
  findMatchingSections(keywords, sectionContents)
  ↓
Calculate coverage percentage
  ↓
formatCoverageReport(report)
  ↓
Exit code: 0 (100%) or 1 (gaps found)
```

## CLI Reference

### Generation

```bash
# Basic generation
/teaching:lecture "Multiple Regression"

# From lesson plan with output directory
/teaching:lecture --from-plan=week05 --output-dir=content/lectures/

# With previous week context and auto-preview
/teaching:lecture "RCBD" --from-plan=week08 --context=previous --open

# Override defaults
/teaching:lecture "ANOVA" --level=graduate --language=python --format=html,pdf

# Force overwrite
/teaching:lecture "RCBD" --from-plan=week08 --output-dir=content/lectures/ --force

# Dry run (preview outline without generating)
/teaching:lecture "Hypothesis Testing" --dry-run
/teaching:lecture "Hypothesis Testing" --dry-run --json
```

### Refinement

```bash
# Refine specific section
/teaching:lecture --refine=content/lectures/week08-rcbd.qmd \
  --section="Worked Examples" \
  --instruction="Add two more examples with R code"

# Full-lecture refinement (no -section)
/teaching:lecture --refine=content/lectures/week08-rcbd.qmd \
  --instruction="Use tidyverse syntax throughout"

# Refine and preview
/teaching:lecture --refine=content/lectures/week08-rcbd.qmd \
  --section="Summary" --instruction="Add key equations" --open
```

### Validation

```bash
# Check coverage against lesson plan
/teaching:lecture --check=content/lectures/week08-rcbd.qmd --from-plan=week08

# Exit code: 0 = 100% coverage, 1 = gaps found
echo $?
```
