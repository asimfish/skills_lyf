## Implementation Waves

### Wave 0: Slide Parser + Type Classifier (~40 tests)

**New file:** `src/teaching/utils/slide-parser.js`

Build on top of existing `qmd-parser.js`:
1. `parseSlides(parsedQmd)` — takes output of `parseQmdContent()`, returns `Slide[]`
2. Each `##` heading = one slide (no `---` separator handling needed)
3. Assign sequential slide numbers
4. Classify type via hybrid cascade:
   - CSS classes (`{.quiz-question}`, `{.section-slide}`, etc.)
   - Heading text patterns ("Quiz:", "Example:", "Practice:")
   - Content heuristics (code blocks, callouts, bullet patterns)
   - Default: `content`
5. Extract metadata: `hasCode`, `hasMath`, `classes[]`
6. Map slide numbers to source line ranges

**Test file:** `tests/teaching/utils/slide-parser.test.js`
- Parse simple deck (5 slides, all ##)
- Type classification from CSS classes
- Type classification from heading patterns
- Type classification from content heuristics
- Default fallback to `content`
- Slide numbering sequential
- Line range mapping correct
- Empty deck handling
- Deck with only frontmatter
- hasCode/hasMath detection

**Dependencies:** None (builds on existing qmd-parser.js)

### Wave 2: Slide Refiner — Targeted Modes (~35 tests)

**New file:** `src/teaching/generators/slide-refiner.js`

Mirrors `lecture-refiner.js` pattern:
1. `reviseSlides(options)` — main entry point
2. `resolveTargets(slides, options)` — normalize `--section`/`--slides`/`--type` to `Slide[]`
3. `reviseSingleSection(slides, targetSlides, options)` — section-targeted revision
4. `reviseSlideRange(slides, targetSlides, options)` — range-targeted revision
5. `reviseByType(slides, targetSlides, options)` — type-targeted revision
6. Adaptive context: full deck if <30 slides, targeted + 1 neighbor if >= 30
7. AI prompt building (section/range/type variants)
8. Write back with provenance metadata

**Modified file:** `src/plugin-api/commands/teaching/slides.md` — Add `--revise` flag docs and `<system>` block

**Test file:** `tests/teaching/generators/slide-refiner.test.js`
- Target resolver: `--section` fuzzy match
- Target resolver: `--slides 5-12` range parsing
- Target resolver: `--type quiz` type filter
- Target resolver: no targets = all slides
- Context strategy: <30 slides sends full deck
- Context strategy: >=30 slides sends targeted + neighbors
- AI prompt includes instruction
- AI prompt preserves slide heading
- Write back preserves frontmatter
- Provenance record appended
- Error: file not found
- Error: no slides matched target

**Dependencies:** Wave 0 (slide parser)

### Wave 3: Bare --revise Auto-Analysis (~20 tests)

**Extend:** `src/teaching/generators/slide-refiner.js`

1. `autoAnalyze(slides, config)` — 7-dimension analysis without AI
2. `buildAutoRevisePrompt(slides, analysis, config)` — prompt from analysis
3. `--dry-run` flag — return analysis report without applying
4. Apply mode — send analysis + slides to AI, get improved deck back

**7 analysis dimensions:**
- Slide density (lines/elements per slide)
- Practice distribution (evenness score)
- Style compliance (check config rules)
- Math depth (formula without explanation)
- Worked examples (key concepts without numerical examples)
- Content completeness (thin slides)
- R output interpretation (code output without interpretation)

**Test file:** `tests/teaching/generators/slide-refiner-auto.test.js`
- Auto-analysis detects overcrowded slide
- Auto-analysis detects sparse slide
- Auto-analysis detects clustered practice slides
- Auto-analysis detects missing math explanation
- Auto-analysis detects R output without interpretation
- `--dry-run` returns report, no file write
- Apply mode writes updated file
- Analysis loads teaching style from config

**Dependencies:** Wave 0 (slide parser), Wave 2 (refiner infrastructure)

### Wave 4: Slide --check Validator (~30 tests)

**New file:** `src/teaching/validators/slide-coverage.js`

Three validation layers:

**Layer 1: Coverage** (reuse `lecture-coverage.js` pattern)
- `validateSlideCoverage(slides, lessonPlan)` — keyword matching across slides in each section
- Extract objectives from lesson plan
- Check coverage across all slides in a `##` section (not per-slide)

**Layer 2: Structure**
- `validateSlideStructure(slides, config)` — ratio checks
- Load expected ratios from `teach-config.yml` `style.structure` section
- Check: slide count, content ratio, practice ratio, quiz per section

**Layer 3: Style**
- `validateSlideStyle(slides, config)` — rule-based checks
- Load rules from `teach-config.yml` `style.validation` section
- Check: math notation, code visibility, callout usage, dtslides classes, hand calculations
- Respect `strictness: advisory | strict`

**Report:**
- `generateSlideCheckReport(results)` — text format with PASS/WARN/FAIL
- `generateReviseCommands(results, filePath)` — ready-to-run `--revise` commands for each issue
- `--json` output option

**Modified file:** `src/plugin-api/commands/teaching/slides.md` — Add `--check` flag docs

**Test file:** `tests/teaching/validators/slide-coverage.test.js`
- Coverage: all objectives found → PASS
- Coverage: some missing → WARN
- Coverage: none found → FAIL
- Structure: ratios within tolerance → PASS
- Structure: practice too low → WARN
- Structure: no quiz slides → WARN
- Style (advisory): missing callout → WARN
- Style (strict): missing callout → FAIL
- Style: math present when config says math: true → PASS
- Style: R output without hand calc → WARN
- Report format: text output matches spec
- Report: includes `--revise` commands for each issue
- JSON output format
- Config missing style section → use defaults

**Dependencies:** Wave 0 (slide parser), existing `lecture-coverage.js` (pattern reference)
